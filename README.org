#+TITLE: tincan.el - a hackish take on agentic coding in Emacs
#+AUTHOR: Marcin 'mbork' Borkowski

This is a small tool to drive Claude Code from Emacs via tmux.  It consists of a few parts:
1. A Python script which takes an ongoing conversation with Claude Code and appends it to a plain text file (watchable with =tail -f= or =auto-revert-tail-mode=).
2. A very simple Emacs mode to show that conversation with font-locking.
3. A slightly more complex Emacs mode to gather the user's responses and send them back to Claude Code.
4. A set of Claude Code hooks to tell Emacs whether it's working or idle.
5. An Emacs command orchestrating all previous parts.

So far everything except the input mode (part 3) is implemented: the transcript
script, the font-locked viewer, the working/idle and needs-input signals, and
the read-only =M-x tincan= command that ties them together.  Sending replies
back to Claude Code (part 3) is not written yet.

* Why both the view and the terminal

# (Terse draft notes; expand into prose.)

- Faithful history
  - from the transcript (JSONL), not the terminal repaint
  - immune to repaint bugs (duplicated/missing lines)
  - main motivation
- Comfortable reply composition
  - dedicated buffer, not the TUI line editor
  - editing directly in vterm/term is awkward
- Free view bonuses: folding, search, rendering, navigation
- Asymmetry between the two surfaces
  - terminal = ground truth + sole input channel
  - view = faithful, navigable reading
  - neither replaces the other
- Caveat: the view is faithful only to what reaches the JSONL
  - TUI-only content absent (e.g. permission prompts)
  - hence the separate "needs-input" hack
- tincan.py stands alone
  - tincan.py + tail -f useful for non-Emacs users too
- Claude Code ToS conformance
  - advantage over tools like agent-shell
  - real Claude Code in a terminal, not a separate API client
  - the official CLI drives Claude

* Input mode - usage

# (Terse draft notes; expand into prose.)

Commands:
- =M-x tincan-start= - run Claude in a vterm, attach a view
  - new session: generates an id, runs =claude --session-id <id>=, shows terminal
  - =C-u M-x tincan-start= - resume (=claude --resume <id>=), shows view, terminal buried
- =M-x tincan-view= - observe a session, read-only
- =M-x tincan-attach= - from a terminal buffer: (re)build and link its view
- =tincan-reply= - compose and send a reply

Keys (view and terminal):
- =C-c SPC= - reply (opens a compose buffer)
- =C-c o= - go to the sibling buffer (view <-> terminal)
- =C-c k= - close the session (ends Claude in the terminal)
- =C-c 0= (view) - delete the terminal window
- =q= (view) - bury the view
- =C-c C-c= (terminal) - send a real interrupt (C-c) to Claude

Compose buffer:
- =C-c C-c= - send to the terminal (bracketed paste + Return)
- =C-c C-k= - cancel

Options:
- =tincan-claude-command= - the program (default =claude=)
- =tincan-show-terminal-on-send= - =display= / =select= / =none= (default =display=)
- =tincan-dismiss-terminal-on-next-command= - next view command deletes the
  terminal window (default =t=)

Requirements and notes:
- vterm needed for the terminal (without it: read-only =tincan-view= only)
- =markdown-ts-mode= (Emacs 31) or =gfm-mode= improves rendering
- flow: view -> reply -> compose -> =C-c C-c= -> terminal -> =C-c o= back

Fix in the existing English text:
- drop "via tmux" (no longer tmux)
- input mode (part 3) is now implemented
- rename =M-x tincan= to =M-x tincan-view=

* Usage

Requirements: Python 3 (standard library only) for the script, and Emacs 30.1+
for the viewer.  For the nicest rendering, install [[https://github.com/jrblevin/markdown-mode][markdown-mode]] (optional).

** Selecting a session

=tincan.py= reads Claude Code's own transcript files under
=~/.claude/projects/=.  Run it from your project (or any subdirectory) to list
that project's sessions, most recent first; add =--all= for every project:

#+begin_src sh
./tincan.py --show-sessions          # sessions of the enclosing project
./tincan.py --show-sessions --all    # every project's sessions
#+end_src

Each line is =SESSION-ID<TAB>TIMESTAMP<TAB>TITLE<TAB>CWD=.  Without =--all= it
lists the sessions of the closest launch directory at or above the working
directory.  A session can be named by its id, a unique id prefix, or a path to a
=.jsonl= transcript.

** Following a transcript with =tail -f=

Print a whole transcript to standard output:

#+begin_src sh
./tincan.py SESSION
#+end_src

Or keep watching a live session and append new turns as they arrive, just like
=tail -f=.  The script only ever appends to stdout and tolerates the partially
written, invalid JSON of an in-progress turn:

#+begin_src sh
./tincan.py SESSION --follow
#+end_src

To watch with =tail -f= itself, stream the transcript into a file and follow
that file:

#+begin_src sh
./tincan.py SESSION --follow > transcript.md &
tail -f transcript.md
#+end_src

** Viewing in Emacs

Load the mode:

#+begin_src emacs-lisp
(add-to-list 'load-path "/path/to/tincan.el")
(require 'tincan)
#+end_src

Then watch a session live with one command:

#+begin_src
M-x tincan
#+end_src

Run it from the project (any subdirectory works).  It lists that project's
sessions, lets you pick one, launches =tincan.py --follow=, and shows the
rendered transcript in a read-only buffer that updates as Claude works.  With a
prefix argument (=C-u M-x tincan=) it offers every project's session instead -
narrow by title or directory - for when you are outside the project tree.

Narrowing by directory in that picker depends on your completion setup: Ivy,
Helm and Vertico (with =orderless=) match anywhere out of the box, while vanilla
=completing-read= wants =flex= or =substring= in =completion-styles= to match a
directory in the middle of a candidate.

The mode line shows the agent's state: =[working]=, =[idle]= (after a =@@@ DONE=
turn-end marker, derived from Claude Code's =turn_duration= record), and - if the
optional hook below is installed - =[needs input]= when Claude is waiting for you
in the terminal.

=M-x tincan= renders through =tincan-render-buffer=, which uses =markdown-mode=
(or =gfm-mode=) when available - including native, per-language fontification of
fenced code blocks - and otherwise falls back to the plain =tincan-view-mode=.
Either way the =@@@ ROLE= section markers (=USER=, =ASSISTANT=, =THINKING=,
=TOOL_USE=, =TOOL_RESULT=, and =DONE=) are font-locked and the buffer is
read-only.

The sections are foldable: =TAB= on a section's =@@@= heading cycles its
visibility, and =S-TAB= cycles the whole buffer.  Every section except =USER= and
=ASSISTANT= starts folded - so thinking, tool calls, tool results and =DONE=
markers stay out of the way until you open them, including as they stream in.
Customize which roles stay open with =tincan-unfolded-sections=.

Markdown headings inside messages fold too: =TAB= on a Markdown heading
(e.g. =##=) folds that heading's subtree; headings nest below the =@@@= sections.

To render a static transcript file instead (for example one you streamed to disk
with =--follow > file=), open it and call =tincan-render-buffer= yourself,
optionally with =auto-revert-tail-mode= for tail-following:

#+begin_src emacs-lisp
(progn
  (find-file "/path/to/transcript.md")
  (tincan-render-buffer)
  (auto-revert-tail-mode))
#+end_src

Relevant options (=M-x customize-group RET tincan RET=):
- =tincan-markdown-mode= - which Markdown mode to use, or =nil= to disable it.
- =tincan-fontify-code-blocks-natively= - per-language code-block highlighting.

** Knowing when Claude wants your input (optional)

When Claude pauses for a tool-permission prompt (or sits idle waiting for a
prompt), nothing is written to the transcript - that prompt is terminal-only
UI - so the viewer cannot show it on its own.  To surface it, tincan can install
an optional Claude Code =Notification= hook that writes a small per-session file
at =<config-dir>/tincan/<session-id>.notify=; =M-x tincan= watches that file and
shows =[needs input]= in the mode line.  The hook only signals that Claude wants
you - you still answer the prompt in the terminal.

Install (and uninstall) it from Emacs:

#+begin_src emacs-lisp
(tincan-install-hook)    ; M-x tincan-install-hook
(tincan-hook-installed-p)
(tincan-uninstall-hook)
#+end_src

These are thin wrappers around the script, which does the actual work and can
also be driven directly:

#+begin_src sh
./tincan.py --install-hook      # also --uninstall-hook, --check-hook
#+end_src

By default the script edits =.claude/settings.local.json= in the current
directory - your personal, gitignored project settings, so the hook fires only
for this project's sessions.  Run install from where you start Claude Code, or
override with =--settings-file= (or, in Emacs, =tincan-hook-settings-file=).  It
backs the file up to =settings.local.json.bak= and keeps any existing settings
and hooks.  Because Claude Code reviews hook changes for safety, restart it or
run =/hooks= afterwards so the hook is loaded.
